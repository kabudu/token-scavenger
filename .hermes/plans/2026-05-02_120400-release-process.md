# Release Process Implementation Plan

> **For Hermes:** Implement this plan task-by-task.

**Goal:** Create a complete release pipeline with GitHub Actions, first-run guided setup, interactive config CLI, and hot-reloadable web UI config changes.

**Architecture:** 
- GitHub workflow: `workflow_dispatch` with semver bump choice, cross-compiles to 3 targets, creates GitHub release
- CLI: extend clap with `config` subcommand + first-run detection in `main.rs`
- Config: add `src/cli/` module for setup wizard and interactive config flows
- Web UI hot-reload: extend existing `PUT /admin/config` to handle server-level settings using `ArcSwap<Config>` already in place, add a config watcher background task

**Tech Stack:** Rust 2024, Clap, GitHub Actions, serde, dialoguer (for interactive prompts), cross-compilation targets

---

### Task 1: Create release workflow (`.github/workflows/release.yml`)

**Objective:** Manual GitHub release workflow with semver bump, cross-compilation, and GitHub release creation.

**Files:**
- Create: `.github/workflows/release.yml`

**Details:**
- Trigger: `workflow_dispatch` with `bump_type` choice (major/minor/patch)
- Steps:
  1. Checkout code
  2. Install Rust toolchain
  3. Install cross-compilation targets (aarch64-apple-darwin, x86_64-pc-windows-msvc, x86_64-unknown-linux-musl)
  4. Parse current version from Cargo.toml
  5. Bump version based on choice
  6. Update Cargo.toml with new version
  7. Build for 3 targets (release)
  8. Create GitHub release with tag `v{new_version}`
  9. Upload binaries as release assets

### Task 3: Implement first-run guided setup

**Objective:** Detect first run, prompt user through config creation, write `~/.config/tokenscavenger/tokenscavenger.toml`.

**Details:**
- Standard config paths (in order): `./tokenscavenger.toml`, `~/.config/tokenscavenger/tokenscavenger.toml`, `~/.tokenscavenger.toml`
- If no config exists and no `--config` flag, enter setup mode
- Guided prompts:
  1. Server bind address (default: 0.0.0.0:8000)
  2. Master API key (optional, hidden input)
  3. Free-first routing? (default: yes)
  4. Provider setup loop: add providers with id, api_key, base_url (optional)
  5. Write file with markers so it's clear this was auto-generated

---

### Task 4: Implement `tokenscavenger config` interactive subcommand

**Objective:** Interactive config editing CLI similar to setup but loads existing config first, shows current values as defaults.

**Details:**
- `tokenscavenger config` subcommand
- Load existing config from detected path or specified `--config`
- Show each section and allow editing:
  1. Server settings (bind, master_api_key, cors)
  2. Providers (list/add/remove/edit)
  3. Routing settings (free_first, paid_fallback)
- After changes, write back to file
- Notify the running instance via HTTP POST to update its runtime config (if running)

### Task 5: Extend web UI config hot-reload to full config schema

**Objective:** Web UI config changes (PUT /admin/config) apply without restart for all config fields, not just provider toggles.

**Files:**
- Modify: `src/api/routes.rs` (extend `admin_config_save` to handle more config fields)
- Modify: `src/app/startup.rs` (add config watch listener background task)

**Details:**
- The `ArcSwap<Config>` + `config_watch_rx` are already in place
- Add a background task in `spawn_background_tasks` that listens on `config_watch_rx` and re-initializes subsystems (route engine, provider registry) when config changes
- `PUT /admin/config` already sends to `config_watch_tx` — extend it to accept full config patches: server.bind, server.master_api_key, routing settings, etc.
- Full config patches write to a `runtime_config.toml` file alongside the main config for persistence
